# ЕГЭ-ХАК · командный центр «Шпиль»

Прод-реализация сигнатурного UX из спецификации (`../ege-hack-project.md`).
Стек: **Next.js 14 (App Router) + TypeScript + Tailwind + framer-motion**, 3D на
**React Three Fiber** (`three` + `@react-three/fiber` + `@react-three/drei`),
состояние — **Zustand**.

## Запуск

```bash
npm install
npm run dev      # http://localhost:3000
```

Прод-сборка:

```bash
npm run build && npm start
npm run typecheck   # проверка типов без сборки
```

> Если нужен просмотр без установки зависимостей — открой
> **`../ege-hack-demo.html`** двойным кликом: автономная версия Шпиля
> (React + Three.js r128 с CDN) с теми же шаблонами.

## Применённые шаблоны

1. **`ui/expandable-tabs.tsx`** — присланный шаблон ExpandableTabs (framer-motion +
   `usehooks-ts`). Использован как переключатель предмета в TopBar. Добавлены
   управляемый режим (`value`) и опция `collapseOnOutsideClick` для навигации.

2. **`ui/liquid-glass.tsx`** — **Liquid Glass** «в тему»: парящие стеклянные
   HUD-панели поверх голографического Шпиля. Реальный `backdrop-filter`,
   спекулярный блик-«линза», преломление под цвет предмета (тёплый Русский /
   холодная Математика) и наклон вслед за курсором. Применён к:
   - консоли готовности (`Console.tsx`, `interactive`),
   - инспектору этажа и карточкам ФИПИ-разбора (`Inspector.tsx`, `sheen`),
   - онбордингу и тосту.

   Почему именно сюда: «Шпиль» — sci-fi командный центр с голограммой в центре.
   Liquid glass усиливает ощущение физического стекла HUD-панелей, преломляющих
   свечение Шпиля — это попадает в дизайн-язык, а не висит украшением.

## Архитектура 3D (порт с ванильного Three.js на R3F)

```
components/spire/
  SpireScene.tsx   — Canvas, камера (наезд/отъезд), свет, звёзды, контролы drag/wheel/pinch
  Floor.tsx        — один этаж: геометрия по типу задания, состояние, дрожь, корона босса
  textures.ts      — процедурные canvas-фактуры (буквы, x², график, кубики…)
  geometry.ts      — геометрия и высоты тел этажей
  PixelBloom.tsx   — пиксельный bloom-переход при «погружении» в тему
lib/
  floor-state.ts   — ghost → forming → unstable → solid
  score-model.ts   — прогноз балла ДИАПАЗОНОМ (этическое требование), вес босса 1.7
  device-tier.ts   — тиринг по мощности (low/mid/high) + reduced-motion
  store.ts         — Zustand: subject / mode / focus / floors
```

## Производительность

3D живёт только на экране Шпиля и грузится лениво (`next/dynamic`, `ssr:false`).
Тиринг по устройству, `pixelRatio` 1–1.5, дрожь — дешёвый CPU-джиттер только на
слабых этажах, уважение к `prefers-reduced-motion` + ручной «лёгкий режим».

## Рост: добавление предметов ЕГЭ — это ДАННЫЕ, не код

Архитектура рассчитана на 15+ предметов. Предмет описывается декларативно, а
визуал (оттенки этажей, геометрия, акценты HUD) выводится билдером.

```
data/
  types.ts          — доменные типы (SubjectDef, TopicDef, Floor, …)
  registry.ts       — РЕЕСТР: массив SubjectDef. Добавить предмет = один объект сюда
  build-subject.ts  — SubjectDef → Subject: hue из палитры (HSL+сдвиг), геометрия
                      по ротации, акценты из baseHue; + validateSubjectDef()
  catalog.ts        — готовое API: SUBJECTS, CARDS, ALL_KEYS, LIVE_KEYS, accentForKey
```

Чтобы подключить, например, Биологию:

```ts
// data/registry.ts — добавить в REGISTRY:
{
  key: "bio", name: "Биология", short: "Биология", exam: "ege",
  goal: 76, maxScore: 100, icon: "Dna", status: "live",
  defaultPattern: "cells",
  theme: { baseHue: 130, sat: 70, light: 58 },   // зелёная палитра
  topics: [
    { id: "bio-cell", name: "Клетка", tag: "цитология", prog: 40, stab: 50 },
    { id: "bio-gen",  name: "Генетика", tag: "наследование", prog: 30, stab: 35 },
    { id: "bio-task", name: "Задача · часть 2", tag: "развёрнутый ответ",
      prog: 0, stab: 0, boss: true, pattern: "scales", crit: [/* … */] },
  ],
}
```

Всё остальное — табы (`TopBar`), пикер онбординга, тема HUD, геометрия и фактуры
этажей — подхватится автоматически. Реестр уже содержит «скоро»-карточки
(Физика, Информатика, Обществознание) как образец.

Добавить **новую фактуру** этажа — одна функция в `spire/textures.ts`; новую
**геометрию** — один `case` в `spire/geometry.ts`; новый ключ в `Pattern`/
`FloorGeom` (types.ts) включит автодополнение и валидацию.

## Решения по фронтенду (senior-обоснование)

- **Почему React/Next, а не Go/Rust.** UI с 3D-Шпилем и анимациями — это React-
  территория: оба присланных шаблона уже на React + framer-motion. Go не делает
  фронтенд; его место — backend-сервис проверки (детерминированная тестовая часть +
  LLM по критериям ФИПИ со структурированным выводом). Rust+WASM дал бы скорость,
  но переписывание R3F и liquid-glass с нуля не окупается. Итог: **Next (фронт) +
  Go-сервис проверки (позже)** — каждый инструмент по сильной стороне.
- **3D изолирован от React.** Сцена грузится лениво (`next/dynamic`, `ssr:false`),
  живёт только на экране Шпиля; цикл практики (`Solve`) намеренно плоский.
- **Состояние — Zustand, селекторами.** Компоненты подписаны на узкие срезы стора,
  не на весь объект — минимум перерисовок при анимации Шпиля.
- **GPU-ресурсы освобождаются.** Геометрии/материалы/буферы диспоузятся при
  размонтировании (важно при частом переключении предметов).
- **Тиринг + reduced-motion + лёгкий режим** — охват дешёвых Android.
- **Liquid glass по делу.** Стекло — язык HUD командного центра (преломляет
  свечение Шпиля под цвет предмета), а не декор.

### Дальше по фронту (бэклог)
- Маршрутизация на App Router (`/spire`, `/solve/[topic]`, `/parent`) вместо
  стейт-свитча — дип-линки и история назад.
- `InstancedMesh` для этажей при большом числе тем; Draco/KTX2 для ассетов.
- Серверные данные предметов (сейчас сид в реестре) → API + калибровка
  «прогноз vs реальный балл».
- Тесты: unit на `score-model`/`floor-state`/`build-subject`, визуальные на экраны.
- a11y-проход: фокус-ловушки в инспекторе, aria для 3D-альтернативы.

> Контент (критерии ФИПИ, примеры) — иллюстративный. Прогноз всегда диапазоном.
